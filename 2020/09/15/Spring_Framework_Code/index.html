Spring 对象管理源码

发表于 2020-09-15 | 分类于 notebook

概述

对上一篇文章【Spring 对象管理】的内容进行补充，主要内容IOC及AOP的源码实现，讲述容器创建过程中，具体做了哪些事情。

IOC

AnnotationConfigApplicationContext，注册配置类相关的实现，用于创建IOC容器，依据参数类型的不同执行不同的逻辑，以下以配置类的实现逻辑为主：

  • this()
    调用无参构造器，此时创建 AnnotationConfigApplicationContext 的对象，同时为部分属性进行赋值，在这个过程中父类（GenericApplicationContext）的无参构造器执行

  • register(componentClasses)
    将当前配置注册进容器

  • refresh()
    容器刷新过程

容器刷新过程

源码如下：

prepareRefresh

容器刷新前的预处理，主要用于容器刷新前为部分属性进行初始化赋值，包括部分环境变量的初始化等操作：

initPropertySources

初始化上下文环境中所需要的部分属性值（主要由子类来实现，父类默认无操作）：

obtainFreshBeanFactory

返回当前容器需要刷新的beanFactory：

prepareBeanFactory

主要是配置工厂的标准上下文特征，例如类加载器和后置处理器，源码如下：

postProcessBeanFactory

Bean Factory 的后置处理，由子类来实现逻辑，父类不提供具体逻辑：

invokeBeanFactoryPostProcessors

调用 BeanFactory 的后置处理器逻辑

PostProcessorRegistrationDelegate.invokeBeanFactoryPostProcessors(..)

第一部分（First）

主要是判断当前 beanFactory 实例是否是 BeanDefinitionRegistry 接口的实例，执行相关逻辑

符合条件执行时：

  • regularPostProcessors
    List<BeanFactoryPostProcessor> regularPostProcessors = new ArrayList<>();
    用于记录 Bean 工厂的后置处理器
  • registryProcessors
    List<BeanDefinitionRegistryPostProcessor> registryProcessors = new ArrayList<>();
    用于记录 Bean 定义注册器的后置处理器

第一步，遍历所有的 Bean 工厂的后置处理器，判断当前处理器的类型是否为 BeanDefinitionRegistryPostProcessor，若是，则先执行处理器的 postProcessBeanDefinitionRegistry 方法，并将对象保存在 registryProcessors 集合中，若不是，则将当前处理器对象保存在 regularPostProcessors 集合中，之后执行。

第二步，按照后置处理器的优先级进行排序并执行处理逻辑，按源码注释顺序

首先，执行实现了 PriorityOrdered 优先级排序接口的 BeanDefinitionRegistry 后置处理器
其次，执行实现了 Ordered 排序接口的 BeanDefinitionRegistry 后置处理器
最后，执行其他的 BeanDefinitionRegistry 后置处理器
每判断一次符合当前条件的处理器时，就执行其相关逻辑，如下：

第三步，执行上面已经记录下来的这些后置处理器的 postProcessBeanFactory 方法，先执行 bean 定义注册后置处理器的逻辑，再执行 bean 工厂后置处理器的逻辑。

不符合条件时：

直接执行当前bean工厂后置处理器的逻辑 invokeBeanFactoryPostProcessors(beanFactoryPostProcessors, beanFactory);。

第二部分（Second）

执行 BeanFactoryPostProcessor 相关的逻辑，逻辑同上，判断从容器中获取到的这些工厂后置处理器的优先执行顺序，按顺序执行其相关逻辑

registerBeanPostProcessors

注册Bean实例创建前的一些拦截用的处理器

initMessageSource

初始化消息资源，国际化之类的，首先判断容器中是否存在 messageSource 实例，如果存在，则直接从容器中获取并赋值，若不存在，则手动创建一个候选的资源实例 DelegatingMessageSource。

initApplicationEventMulticaster

初始化事件派发器（多播器），先判断容器中是否存在 applicationEventMulticaster 实例，存在则赋值，不存在则创建一个简单的事件派发器 SimpleApplicationEventMulticaster 实例。

onRefresh

在特定上下文子类中初始化其他特殊bean。

registerListeners

注册监听器，若存在一些早期时间需要进行处理的，会将这些早期时间进行推送，然后监听器监听到这些事件后会执行相应逻辑

处理早期事件，执行方式有两种，存在任务执行器则通过执行器进行事件监听方法的回调，如果不存在，则直接回调事件的监听方法

finishBeanFactoryInitialization

完成bean工厂实例的初始化工作，上面所有的步骤都可以看成是在对象在初始化之前要做的准备

beanFactory.preInstantiateSingletons

单实例对象的实例化过程，包括对象创建，属性注入，初始化等操作，在这些操作内部也穿插者一些列的处理器相关的逻辑，如对象初始化前后的后置处理器逻辑，生成代理对象的相关逻辑。

若当前对象非抽象类，非懒加载组件，且是单实例对象，则进一步判断是否为 FactoryBean 的实现，若是则后续通过 getObject() 的方式获取对象实例，如果不是，则从容器缓存中直接获取对象实例。

核心逻辑：

createBean
resolveBeforeInstantiation

对象实例化之前进行解析，主要用于给后置处理器生成一个代理对象，用于替换原来的对象实例，主要通过判断后置处理器是否实现了 InstantiationAwareBeanPostProcessor接口

执行实例化之前的后置处理逻辑时，判断当前处理器是否为 InstantiationAwareBeanPostProcessor 接口的实现，若是，则调用他的逻辑，并返回一个对象，如果对象不为空，则会调用其后置处理器中初始化后的逻辑。

如果当前后置处理器实现了 InstantiationAwareBeanPostProcessor 接口，则会调用如下：

doCreateBean

创建实例：createBeanInstance(beanName, mbd, args);

准备Bean：populateBean(beanName, mbd, instanceWrapper);

初始化：initializeBean(beanName, exposedObject, mbd);，分为如下几个步骤：

先调用实现了Aware接口的相关方法：invokeAwareMethods(beanName, bean);

执行Bean的后置处理器前置操作：applyBeanPostProcessorsBeforeInitialization(wrappedBean, beanName);
执行初始化方法：invokeInitMethods(beanName, wrappedBean, mbd);

执行Bean的后置处理器后置操作：applyBeanPostProcessorsAfterInitialization(wrappedBean, beanName);

这个接口实现了上面所说的 InstantiationAwareBeanPostProcessor 接口
