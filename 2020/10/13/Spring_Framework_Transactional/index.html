Spring 事务管理

发表于 2020-10-13 | 分类于 notebook

概述

本文主要对数据库的事务进行一个介绍，包括，事务特性、事务属性、事务传播行为等，旨在程序开发过程中，如何声明一个事务，如何对事务进行优化等。

事务特性（ACID）

事务的基本基本特性主要分为如下四种：

  • 原子性（Atomic）
    事务操作逻辑，必须全部成功完成，一个出错，则整个事务回滚
  • 一致性（Consistence）
    事务操作的结果必须和事务操作前的总量一致
  • 隔离性（Isolation）
    事务与事务之间相互隔离
  • 持久性（Durability）
    事务一旦提交，那么修改就是永久的

事务属性

事务的超时属性（timeout）

通过在需要添加事务方法上添加 @Transactional 注解，并设置超时（timeout）属性值（设置的 3s）。

为事务添加超时属性，若事务超时未完成，则会抛出事务超时异常 org.springframework.transaction.TransactionTimedOutException。

事务的只读属性（readOnly）

用于实现事务优化，对一些只读操作提高其查询效率。

事务回滚属性（rollback）

事务回滚，默认只针对非受检异常（或，运行时异常），对于受检异常（或，编译时异常），如果需要实现事务回滚则需要配置回滚属性，如下：

  • rollbackFor
    对于哪些异常需要回滚，能够接受一个异常类型数组（Class<? extends Throwable>[]）
  • rollbackForClassName
    功能同上，只不过接收类型为一个字符串数组，值为对应异常的全限定类名（String[]）
  • noRollbackFor
    指定哪些类型的异常不进行事务回滚，接收一个异常类型数组（Class<? extends Throwable>[]）
  • noRollbackForClassName
    功能同上，只不过接收类型为一个字符串数组，值为对应异常的全限定类名（String[]）

事务隔离级别（isolation）

隔离级别枚举类：

  • 脏读
    最应该避免的错误读取，指在两个事务同时进行的过程中，事务一进行读取，事务二进行修改
    事务二修改了数据之后（未提交），事务一读取到了最新修改的数据
    事务二进行了数据回滚，此时事务一读取到的新数据就成了脏（无效）数据，如果用这个数据进行后续的逻辑操作，绝对是错误的。

  • 不可重复读
    不可重复读是指在事务一内，读取了一个数据，事务一还没有结束
    事务二访问并修改了这个数据，并提交
    事务一又读这个数据，由于事务二的修改，那么事务一两次读到的的数据可能是不一样的

  • 幻读
    所谓幻读，指的是当某个事务在读取某个范围内的记录时，另外一个事务又在该范围内插入了新的记录，当之前的事务再次读取该范围的记录时，会产生幻行。InnoDB存储引擎通过多版本并发控制（MVCC）解决了幻读的问题
    事务一查询某张表的范围内数据
    事务二在数据表中新增了部分数据
    事务一再次读取相同范围的数据时，发现多了一些之前没有的数据

如下表：各隔离级别可能出现的问题

隔离级别 描述 脏读 不可重复读 幻读 加锁读
READ-UNCOMMITTED 读未提交 Y Y Y -
READ_COMMITTED 读已提交 - Y Y -
REPEATABLE_READ 可重复读 - - Y -
SERIALIZABLE 串行化 - - - Y

其他

MySQL，使用如下命令查看当前会话的事务隔离级别：
SELECT @@transaction_isolation（早期版本：SELECT @@tx_isolation）

使用如下命令设置当前会话的事务隔离级别：
set session transaction isolation level 事务隔离级别
